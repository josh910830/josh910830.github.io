계산기의 원리가 궁금해서 시작한 공부가 디자인패턴의 아름다움을 만나 직업이 되었습니다. 1년 반 동안 스타트업에서 조직의 성장과 협업의 변화를 겪으며 개발생산성을 고민하였습니다. 테스트와 아키텍처의 베스트 프랙티스를 탐색하며 지속가능한 코드를 작성하려 노력합니다.

뛰어난 기술자, 그 이상으로 ‘좋은 동료’가 되고 싶습니다.

기술

자바, 스프링, JPA를 주로 사용하며 객체지향, 관계형모델, HTTP 등 기반의 이해를 중시합니다. 카프카 및 다양한 NoSQL을 업무적으로 사용하였고 최적의 사용법을 꾸준히 고민하고 있습니다. 클라우드 네이티브에 흥미가 있으며 애자일과 오너십의 관점으로 기술들을 학습하고 있습니다.

  • Java Kotlin Spring JPA Gradle OptaPlanner
  • Postgresql Cassandra Redis ElasticSearch Kafka
  • Linux Docker AWS(EC2/RDS/Lambda) Git/Github Slack etc.
  • Javascript React Redux HTML/CSS : 개인 프로젝트 수준에서 경험하였습니다.
  • Kubernetes ArgoCD Grafana : 사내에 구성된 인프라를 사용, 배포 및 모니터링하였습니다.

테크타카

물류서비스 ARGO 창고관리시스템 백엔드엔지니어 / 2021. 7 ~ 2023. 2

집품 단위 주문 재구성

창고 내 작업 효율의 개선을 위해 주문의 상품들을 집품 단위로 재구성하는 기능입니다.

초기 스타트업으로서 고객의 니즈를 완전히 파악할 수 없는 상황이었기에 최대한 유연한 설계를 필요로 했고 도메인 모델링이 까다로운 상황에서 조건 함수 자체를 데이터베이스에 저장하는 아이디어를 제시하였습니다.

룰 엔진을 리서치하였으나 유지보수 시 학습비용을 고려하여 보다 직관적인 스프링 표현식을 선택하였고 테스트 주도 개발을 진행하여 빠른 시점에 엣지 케이스들을 발견하며 기획 정교화에 기여할 수 있었습니다.

마이크로 서비스 아키텍처

사업적으로 주문관리시스템(OMS)과 창고관리시스템(WMS)을 분리하며 MSA로의 전환이 시작되었습니다. 사전 작업으로 상호 컴포넌트에의 직접 의존 제거를 담당하였고 어댑터 패턴을 활용해 OCP를 준수하였습니다.

한편 기존 메인 DB로 사용하던 카산드라가 파티션이라는 구조적 제약으로 기능 추가에 어려움을 주고 있었기에 포스트그레스로 DB를 변경하게 되었고, 이에 따라 WMS 도메인 모델의 재설계 초안을 작성하였습니다.

신규 리포지토리로 이전하면서 코드 품질을 개선하고자 사내 상태머신 프레임워크의 사용 컨벤션을 제안하였고 순환의존 제거를 위해 스프링 이벤트를, 반복코드 제거를 위해 AOP와 JPA Audit을 도입했습니다.

기존 애플리케이션을 프록시로 사용하여 스위치로 배포와 롤백을 진행하는 방식을 경험해보기도 하였습니다.

작업 추천 서비스

상품을 적재/집품할 위치를 선정하고 작업자에 맞게 분할하는 WMS 내 마이크로 서비스입니다. 집합 덮개 문제(NP)를 해결하며 제한시간 내에 차선의 해답을 찾는 플래닝 엔진(OptaPlanner)을 사용합니다.

기술의 특이성으로 인해 전담 인력이 존재해왔으나 기존 담당자가 매니저를 수행함에 따라 인계받게 되었습니다. 유연한 인력 운용을 위해 진입장벽을 낮추어야 한다고 생각해 외부 엔진을 의존성 역전 방식으로 격리하였습니다.

한 단계의 정책이 설정에 따라 다중화되는(이력기반 위치선정 실패 시 공간최적화 위치선정) 기능이 추가되며 분기처리 시 조합폭발이 발생할 수 있었으므로 컴포지트 패턴을 응용하여 함수 합성으로 구현하였습니다.

큐의 요청을 쉬고있는 인스턴스가 스케줄링으로 폴링하여 처리하고 있었기에 바쁜 시간대에 지연이 발생했었으나 폴링 트리거를 API로 변경하고 오토 스케일을 적용, AWS 람다로 스케줄링하도록 수정하여 문제를 해결했습니다.

운영 알림 발송

출고 통계와 재고 현황에 따른 입고 추천 등 운영부서에 알림을 발송하는 기능입니다.

첫 외부 물류센터에 시스템을 제공하는 시점에서 알림 채널 관리의 필요성을 건의하여 개발하게 되었습니다. 사내의 슬랙 피로도를 최소화하고 싶었고 너무 많은 알림이 중요한 알림을 놓치게 하는 위험을 막고자 했습니다.

발송 인터페이스와 플랫폼(메일/슬랙)별 구현을 분리하여 자체 CC 및 플랫폼별 포매팅을 빠르게 작업하였습니다. 알림 내용의 보완과 함께 채널 전략에 관해서도 다양한 부서와 협의함으로써 운영 상황 인지에 보탬이 되었습니다.

발생하는 운영이슈들에 유관부서와 논의하여 현장 가이드라인을 도출하기도 하였습니다.
